# Phase 1 — 要件定義・仕様策定インタビュー

## 概要

このスキルは以下の流れで実行し、Phase 2 以降がエージェント自律実行できる状態を作る。

```
Step 0   — 開始宣言
Step 0-R — 参考資料の読み込み・情報収集（docs/refs/ + 外部URL）
─── パート A: インタビュー ───
  Step A-1   — プロジェクト基本情報
  Step A-1-G — ゴール設定の明確化（専用深掘りステップ）
  Step A-2   — ターゲット・ユーザー
  Step A-3   — 機能要件・非機能要件
  Step A-4   — デザイン・UX（UI がある場合）
  Step A-5   — 技術的制約・環境
─── パート B: 技術スタック提案・合意 ───
  Step B-1 — 分析と提案
  Step B-2 — 合意確認
─── パート C: ドキュメント生成 ───
  Step C-1 — docs/requirements.md
  Step C-2 — docs/spec/basic.md
  Step C-3 — docs/spec/detail.md
  Step C-4 — docs/phases.md
  Step C-5 — 完了メッセージ
```

---

## 禁止事項

- ユーザーの回答を待たずに次のカテゴリへ進まない
- ゴール設定ステップ（Step A-1-G）・参考資料ステップ（Step 0-R）を省略しない
- 確認ステップ（各タスク末尾の合意取得）を省略しない
- `docs/requirements.md`・`docs/spec/basic.md`・`docs/spec/detail.md`・`docs/phases.md` 以外のファイルを生成しない
- `docs/refs/` 内のファイルは読み取りのみ行い、編集・削除しない
- git 操作（コミット・プッシュ）は行わない

---

## 実行手順

---

### Step 0 — 開始宣言

以下のメッセージを表示してユーザーの準備を確認する：

```
# Phase 1 — 要件定義・仕様策定インタビュー

カテゴリごとに質問します。すべての回答が揃ったら以下のファイルを生成します：
- docs/requirements.md（要件定義書）
- docs/spec/basic.md（基本仕様書）
- docs/spec/detail.md（詳細仕様書）
- docs/phases.md（工程計画）

【参考資料がある場合】
事前に docs/refs/ フォルダに資料（PDF・Markdown・テキスト等）を置いておくと、
インタビュー前に内容を読み込んで活用します。

準備ができたら「始めて」と入力してください。
```

ユーザーの応答を待つ。

---

### Step 0-R — 参考資料の読み込み・情報収集

#### Step 0-R-1: docs/refs/ の確認

`docs/refs/` ディレクトリ内のファイルを確認する。

**ファイルが存在する場合：**
- すべてのファイルを読み込む（PDF・Markdown・テキスト・画像等）
- 読み込んだ内容を以下の観点で整理する：
  - プロジェクト名・目的・背景
  - 要件・機能のヒント
  - ゴール・KPI の記述
  - 技術・デザインに関する制約や方向性
  - 参考にすべきサービス・サイト・競合情報

読み込み完了後、以下を表示する：

```
## 参考資料を読み込みました

以下の資料を確認しました：
- [ファイル名]: [内容の要約1〜2行]
- ...

これらの情報をインタビューに活用します。
```

**ファイルが存在しない場合：** その旨を記録してスキップし、Step 0-R-2 へ進む。

---

#### Step 0-R-2: 外部 URL・追加資料の確認

以下を表示してユーザーの入力を待つ：

```
## 参考 URL・追加資料の確認

参考にしたいウェブサイト・競合サービス・既存システムの URL があれば教えてください。
（複数ある場合は改行して入力してください）

なければ「なし」と入力してください。
```

**URL が提供された場合：**
- 各 URL を WebFetch ツールで取得し、以下を抽出する：
  - サービス・サイトの目的・特徴
  - UI 構成・ページ構成
  - 機能・コンテンツの種類
  - デザイントーン・カラー傾向
- 抽出結果を簡潔にまとめて表示する：

```
## 参考サイト分析結果

### [URL またはサイト名]
- 目的・特徴: ...
- 構成: ...
- 参考にできる点: ...
```

**「なし」の場合：** スキップしてパート A へ進む。

---

## ─── パート A: インタビュー ───

> **注意**: Step 0-R で取得した情報で回答が明らかな質問は「参考資料から〇〇と読み取りましたが、合っていますか？」と確認形式にしてよい。

---

### Step A-1 — プロジェクト基本情報

```
## Q1. プロジェクト基本情報

1. プロジェクト（システム・サービス・サイト）の名前は何ですか？
2. クライアント（依頼主）はどなたですか？個人・法人どちらですか？
3. このプロジェクトの目的・解決したい課題を教えてください。
4. Notion 等で管理しているプロジェクトページの URL があれば教えてください。
```

回答を受け取ったら、曖昧な点があれば1〜2問だけ深掘りして確認する。
その後、**Step A-1-G（ゴール設定）に必ず進む**。

---

### Step A-1-G — ゴール設定の明確化

このステップの目的は、「達成したいゴール」を**測定可能で優先順位のついた形**に落とし込むことである。
曖昧な回答（「集客したい」「便利にしたい」）が出た場合は、必ず具体的な数値・状態・期限に変換する。

```
## Q1-G. ゴール設定

プロジェクトの成功イメージを具体化します。

1. このプロジェクトが「成功した」と言えるのは、どんな状態になったときですか？
   （例：「毎月○件の問い合わせが来る」「作業時間が週○時間削減できた」「○人が使ってくれている」）

2. リリースから 1ヶ月後・3ヶ月後・1年後に、それぞれどんな状態を目指しますか？

3. 数値で測れる目標（KPI）があれば教えてください。
   （例：月間PV・コンバージョン率・処理件数・エラー率・ユーザー数）

4. ゴールが複数ある場合、優先順位を教えてください。
   （何かトレードオフが発生した場合、何を優先しますか？）

5. 逆に、これだけは避けたい・失敗とみなす状態はありますか？
```

**回答を受け取ったら以下を実施する：**
- 「○ヶ月後に△△な状態」という形で整理して提示する
- 数値が出ていない場合は「具体的な数値で表すとどのくらいですか？」と深掘りする
- ゴールが抽象的なまま残らないよう、必ず測定可能な形に変換して合意を取る

**確認フォーマット：**

```
## ゴール確認

以下の形でゴールを整理しました。これで合っていますか？

### 最重要ゴール（最優先）
[具体的・測定可能な形で記述]

### サブゴール
- [期限]: [状態]
- [期限]: [状態]

### KPI（測定指標）
| 指標 | 目標値 | 測定タイミング |
|---|---|---|
| | | |

### 避けたい状態（失敗の定義）
[記述]

問題なければ「OK」と入力してください。
```

ユーザーが承認したら **Step A-2** へ進む。

---

### Step A-2 — ターゲット・ユーザー

```
## Q2. ターゲット・ユーザー

1. このシステム・サービスのメインのターゲット層を教えてください（属性・役割・スキルレベルなど）。
2. ユーザーがこのシステムを使う主な場面・動機は何ですか？
3. 主な利用環境を教えてください（例：PC・スマホ・タブレット・社内ネットワーク限定 など）。
```

---

### Step A-3 — 機能要件・非機能要件

```
## Q3. 機能要件・非機能要件

1. 絶対に必要な機能・画面を教えてください。
2. あると嬉しいが必須ではない機能はありますか？
3. 「やらない・実装しない」と明確に決めていることはありますか？
4. パフォーマンス・セキュリティ・アクセシビリティ・多言語対応などの非機能要件はありますか？
5. 既存システムやデータとの連携が必要ですか？
```

---

### Step A-4 — デザイン・UX（UI が発生する場合）

UI・画面を持つシステムの場合のみこのカテゴリを実施する。
UI を持たないシステム（バッチ処理・API・CLI 等）の場合はスキップし、その旨をユーザーに伝える。

```
## Q4. デザイン・UX（UI がある場合）

1. ブランドカラー・ロゴはありますか？カラーコードや色のイメージを教えてください。
2. 全体の雰囲気・トーンを教えてください（例：シンプル・高級感・ポップ・業務的 など）。
3. 参考にしたいデザイン・既存サービスがあれば教えてください。（Step 0-R で取得済みのものは省略可）
4. 既存の素材（ロゴ・写真・アイコン）はありますか？
```

---

### Step A-5 — 技術的制約・環境

```
## Q5. 技術的制約・環境

1. 使用を希望する、または禁止されている技術・言語・フレームワークはありますか？
2. 実行環境・ホスティング先の希望はありますか？（例：GitHub Pages・AWS・社内サーバー・Vercel）
3. 開発チームの人数・スキルセットを教えてください。
4. 公開・リリースの希望日はありますか？
5. 運用・保守体制はどうなりますか？（例：エンジニアが常駐 / ノーメンテを希望 / 定期更新あり）
6. コーディング規約・命名規則・コードスタイルに指定はありますか？
   （例：コメントは日本語・インデントはタブ・既存コードベースの規約に合わせる等。
   　特になければスキップします）
```

## ─── パート B: 技術スタック提案・合意 ───

### Step B-1 — 技術スタックの分析と提案

パート A の回答と Step 0-R で収集した参考情報を踏まえ、以下の観点でシステム種別を判定し、最適な技術スタックを 2〜3 案提案する。

**判定ポイント：**
- UI の有無（WEB・モバイル・デスクトップ・CLI・API）
- 動的機能の要否（静的 / サーバーサイド / フルスタック）
- 運用・保守制約（ノーメンテ志向 / 定期更新あり）
- チームのスキルセット
- ホスティング・インフラ制約
- ゴール・KPI に対する適合性

**提案フォーマット：**

```
## 技術スタック提案

プロジェクトの特性を踏まえ、以下の案を提案します。

### 案 1: [名称]（推奨）
- フロントエンド: ...
- バックエンド: ...
- DB / ストレージ: ...
- ホスティング: ...
- 選定理由: ...
- 懸念点: ...

### 案 2: [名称]
...

### 案 3: [名称]（シンプル重視）
...

どの案が適していますか？変更・追加したい点があれば教えてください。
```

ユーザーの回答・フィードバックを受け取り、必要に応じてカスタマイズして合意を取る。

```
|---|---|---|
| ... | ... | ... |

問題なければ「OK」と入力してください。
```

ユーザーが承認したら パート C へ進む。

### Step C-4 — docs/phases.md の生成

パート A・B の回答（特に技術スタック・機能一覧・スケジュール・ゴール）を踏まえ、`docs/phases.md` をプロジェクトに合わせた形で生成する。

**フェーズ構成の設計指針：**
- **Phase 1（要件定義・仕様策定）と最終検証フェーズは必須**。必ずフェーズ構成に含めること
- Phase 2 はプロトタイプ・サンプル・MVP の作成
- 中間フェーズは機能の実装・調整
- 最終の1つ前フェーズはデプロイ・公開
- **最終フェーズは必ず「最終検証」とし、以下を含める：**
  - 成果物の動作確認（仕様書との照合）
  - `docs/requirements.md` のゴール・KPI に対する達成確認
  - 納品・リリース判定
- フェーズ数はプロジェクト規模に応じて 4〜8 程度で設計する（Phase 1 と最終検証を含む）
- **ゴール・KPI の達成に必要なフェーズが漏れていないか確認する**

**出力フォーマット：**

```markdown
# [プロジェクト名] — 工程定義

## Phase 1: 要件定義・仕様策定【完了】

**目標**: プロジェクトの要件・仕様・工程を確定し、エージェントが動作できる土台を整える。

| タスク | 状態 |
|---|---|
| 参考資料の収集・分析（docs/refs/） | 完了 |
| 要件定義・ゴール設定（docs/requirements.md） | 完了 |

| 工程計画策定（このファイル） | 完了 |

**目標**: ...

| タスク | 状態 |
```

### Step C-5 — 完了メッセージ

```
## Phase 1 インタビュー完了

以下のファイルを生成しました：

- docs/requirements.md  — 要件定義書（ゴール・KPI・参考資料含む）
- docs/spec/basic.md    — 基本仕様書
- docs/spec/detail.md   — 詳細仕様書
- docs/phases.md        — 工程計画

次のステップ：
1. 各ファイルの内容を確認・修正してください。
2. 問題なければ `git add docs/ && git commit` で保存してください。
3. Phase 2 以降は `/run-phase 2` で自律実行できます。
```
