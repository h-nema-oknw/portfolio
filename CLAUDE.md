# CLAUDE.md

Claude Code エージェント駆動開発フレームワークの設定ファイル。
このファイルは **フレームワーク共通テンプレートとして固定** する。
プロジェクト固有の情報はすべて `docs/` 配下のファイルに格納し、このファイルには記載しない。

---

## プロジェクト仕様の参照先

エージェントはプロジェクト情報を以下のファイルから取得する。

| ファイル | 内容 | 生成タイミング |
|---|---|---|
| `docs/requirements.md` | 要件定義書（要件・非要件） | Phase 1 |
| `docs/spec/basic.md` | 基本仕様書（目的・構成・機能概要） | Phase 1 |
| `docs/spec/detail.md` | 詳細仕様書（技術スタック・仕様値） | Phase 1 |
| `docs/phases.md` | 工程計画・タスク一覧・進捗 | Phase 1 |
| `config/git-policy.md` | Git コミットポリシー設定 | Phase 1（変更: `/git-policy`） |

---

## ディレクトリ構成と役割

```
README.md                  プロジェクト固有の説明（自由記述・エージェント編集禁止）
help.md                    FORGEフレームワーク使用ガイド（編集禁止・/forge-help で参照）

agents/
    criteria.md            評価指標・合格スコア定義（編集禁止）
    phases/
        phase{N}/
            plan.md        エージェントが作成する実施方針・構成（エージェントが生成）
            task.md        エージェントが作成する具体的タスク指示（エージェントが生成）

docs/
    requirements.md        要件定義書（編集禁止）
    spec/
        basic.md           基本仕様書（編集禁止）
        detail.md          詳細仕様書（編集禁止）
    phases.md              工程計画・進捗管理（Step 6 の状態更新のみ可）
    refs/                  参考資料置き場（PDF・MD・画像・URLリスト等。編集禁止・読み取り専用）

config/
    git-policy.md          Git コミットポリシー設定（/git-policy スキルで管理）

result/                    成果物の置き場所
reports/                   各フェーズのループ完了後レポートの出力先
```

## ブランチ保護ルール

**エージェントは作業開始前に必ず現在のブランチを確認すること。**

- `git branch --show-current` でブランチ名を確認する
- ブランチ名が `main` だった場合は、**いかなる処理も実行せず即座に中断する**
- 中断時はユーザーに以下のメッセージを表示すること：
  > 現在 `main` ブランチ上にいます。作業用ブランチを切ってから再実行してください。  
  > 例: `git checkout -b phase{N}-work`

## git 操作ルール

エージェントの git 操作は **`config/git-policy.md`** の「操作レベル」に従う。

**作業開始前に必ず `config/git-policy.md` を読み込み、操作レベルを確認すること。**
ファイルが存在しない場合はレベル 0（禁止）として扱う。

### レベル 0 — 禁止（ファイルが存在しない場合も同様）

- git 操作は一切行わない
- ファイルの作成・編集のみ行う

### レベル 1 — コミットのみ（試験運用）

- Step 7（引き継ぎ記述）完了後にコミットを実行する
- コミット対象：`result/`・`agents/phases/phase{N}/`・`reports/phase{N}_report.md`・`docs/phases.md`
- コミットメッセージ形式：`phase{N}: [フェーズ名] 完了`
- プッシュ・PR 作成は行わない（人間が手動で実施する）

### レベル 2 — PR 作成まで（本運用）

- Step 7 完了後にコミット → 即プッシュ（現在の作業ブランチのみ）
- コミット対象・メッセージ形式：レベル 1 と同様
- 最終フェーズの全 Step 完了後に PR を作成する（`gh pr create --base main --head [現在のブランチ名]`）
- **main への直接プッシュは禁止**
- **force push は禁止**
- **ブランチ操作（切り替え・作成・削除）は禁止**

## エージェントの自律動作フロー

エージェントは以下の順序で自律的に動作する。工程全体の進捗は [docs/phases.md](docs/phases.md) を参照すること。

---

### Step 1 — 前フェーズの引き継ぎ内容の読み取り

- 直前フェーズの `reports/phase{N-1}_report.md` が存在する場合は必ず読み込む
- 次フェーズへの引き継ぎ事項・懸案事項を把握し、以降のステップに反映する
- 初回フェーズ（Phase 2）の場合はスキップしてよい

---

### Step 2 — プランの作成

- `docs/phases.md` を参照し、現在進行中のフェーズの定義・タスク一覧を把握する
- `docs/requirements.md`・`docs/spec/basic.md`・`docs/spec/detail.md` を読み込み、仕様を把握する
- そのフェーズに対応する `agents/phases/phase{N}/plan.md` を作成する
- `plan.md` には、`task.md` に落とし込む前の実施方針・構成・注意点をまとめる
- Step 1 で把握した引き継ぎ事項を計画に反映すること
- 既に `plan.md` が存在する場合は内容を確認してから上書き判断すること

---

### Step 3 — プランの自己評価

- `agents/criteria.md` の Stage 1（プラン評価）基準に従い、作成した `plan.md` を自己評価する（100点満点）
- 評価結果・スコア・コメントを記録する

#### Step 3-1 — 不合格（80点未満）の場合

- 指摘箇所と改善方針を明確にしてから `plan.md` を修正する
- 修正後、Step 3 に戻り再評価する
- 同じ指摘を3回修正してもスコアが改善しない場合は、プランのアプローチを根本から見直す
- 最大5回の試行を超えた場合は現時点のプランと課題をレポートに記録して処理を終了する

#### Step 3-2 — 合格（80点以上）の場合

- `plan.md` の内容を基に、`agents/phases/phase{N}/task.md` を作成する
- `task.md` には、エージェントが迷わず実行できる具体的な手順・指示を記載する

---

### Step 4 — タスクの実行

- `agents/phases/phase{N}/task.md` の指示に従い、`result/` 配下に成果物を作成・編集する
- `docs/spec/detail.md` の仕様値を正確に参照しながら実装する

---

### Step 5 — 成果物の自己評価

- `agents/criteria.md` の Stage 2（成果物評価）基準に従い、生成した成果物を自己評価する（100点満点）
- 評価結果・スコア・コメントを記録する

#### Step 5-1 — 不合格（80点未満）の場合

- 指摘箇所と改善方針を明確にしてから成果物を修正する
- 修正後、Step 5 に戻り再評価する
- 同じ指摘を3回修正してもスコアが改善しない場合は、実装アプローチを根本から見直す
- 最大5回の試行を超えた場合は現時点のスコアと課題をレポートに記録して処理を終了する

#### Step 5-2 — 合格（80点以上）の場合

- 実行内容・最終スコア・評価コメント・成果物パスを `reports/phase{N}_report.md` に出力する
- レポートには次フェーズへの引き継ぎ事項を含めること

---

### Step 6 — phases.md の更新

- `docs/phases.md` の対象フェーズ内の各タスクの状態を実施結果に合わせて更新する
- 完了したタスクは「完了」、次フェーズに持ち越すものは「未着手」のまま残す
- `docs/phases.md` はこの更新操作のみ例外的に書き込みを許可する（他の `docs/` ファイルは読み取り専用）

---

### Step 7 — 次フェーズへの引き継ぎ内容の記述

- `reports/phase{N}_report.md` の末尾に、次フェーズの担当エージェントへ向けた引き継ぎセクションを追記する
- 記述すべき内容：
  - 今回のフェーズで実施した内容のサマリー
  - 未解決の課題・懸案事項
  - 次フェーズで注意すべき点・前提条件
- セッションをまたいでも意図が伝わるよう、背景情報を省略せずに記述すること
- **引き継ぎ記述完了後、`config/git-policy.md` の操作レベルに従って git 操作を実行する**
  - レベル 0：何もしない
  - レベル 1：コミットを実行する（`git add` → `git commit -m "phase{N}: [フェーズ名] 完了"`）
  - レベル 2：コミット → プッシュを実行する（`git push origin [現在のブランチ名]`）

---

### Step 8 — Notion への進捗反映

- `docs/requirements.md` に記載された Notion ページ URL（または Notion ページ名）を参照する
- 該当ページが見つからない場合は Notion 内でプロジェクト名を検索して特定する
- 以下を反映する：
  - フェーズの完了ステータス更新
  - 実施内容のサマリー
  - 次フェーズに向けた懸案事項・引き継ぎ事項（ある場合）
- Notion の既存構成・フォーマットを尊重し、不要な構造変更はしない
- **これが最終フェーズかつ操作レベルが 2 の場合、Step 8 完了後に PR を作成する**
  - `gh pr create --base main --head [現在のブランチ名] --title "[プロジェクト名] 実装完了" --body "[フェーズ一覧と実施内容のサマリー]"`

## トークン管理ルール

各 Step の完了後、次の Step に進む前に残りトークン量を確認し、以下の基準で継続可否を判断する。

| 次の Step | 最低必要トークン（目安） |
|---|---|
| Step 2（plan.md 作成） | 8,000 |
| Step 3（プラン評価ループ） | 10,000 |
| Step 4（タスク実行） | 20,000 |
| Step 5（成果物評価ループ） | 10,000 |
| Step 6〜8（更新・引き継ぎ・Notion） | 5,000 |

- 残りトークンが次 Step の必要量を下回る場合は、**その Step を実行せず中断する**
- 残りトークンが少ない場合でも Step 6〜8 は軽量なため、可能な限り実行を試みる

### 中断時の処理

1. `reports/phase{N}_resume.md` を出力する（次回セッションの再開用プロンプトとして機能させる）
2. **git 操作はトークン不足による中断時は行わない**（ファイル出力のみ。コミットは次回セッションで再開後に実施する）

`phase{N}_resume.md` に含める内容：

- 中断した Step 番号と理由（トークン不足による中断）
- 現在の進捗状態（完了済みの Step、生成済みファイルのパス）
- 次回セッションへの指示プロンプト（コピー＆ペーストでそのまま使えるよう記述する）
- 未完了タスクの一覧

## ファイル操作のルール

- 成果物ファイルは必ず `result/` ディレクトリ配下に作成・編集すること
- `agents/` 配下のファイルは読み取り専用とし、編集しないこと（`plan.md` / `task.md` の作成を除く）
- `docs/requirements.md`・`docs/spec/basic.md`・`docs/spec/detail.md`・`docs/refs/`・`agents/criteria.md` はエージェントによる編集を禁止する（読み取り専用）
- 上記ファイルは Phase 1（`/phase1-interview`）でのみ生成・編集を許可する。Phase 2 以降のエージェントは一切書き込まないこと
- `docs/phases.md` は Step 6 のフェーズ状態更新のみ例外とする
- `reports/` への書き込みはレポート出力時のみ行うこと
- `config/git-policy.md` は Phase 1（`/phase1-interview`）で生成し、`/git-policy` スキルでのみ更新を許可する

---

## 実装のルール

- `docs/spec/detail.md` の仕様値（値・設定・技術選定）を正確に参照し、それに従って実装すること
- 仕様書（`docs/spec/basic.md`・`docs/spec/detail.md`）に記載のない機能は一切実装しないこと
- プロジェクト固有の技術制約は `docs/spec/detail.md` の「技術スタック」セクションに従うこと
- 外部リソース・サービスを使用する場合は `docs/spec/detail.md` に記載のものに限定すること

---

## フェーズ構成

- フェーズ構成は **プロジェクトごとに動的に決まる**。固定ではない。
- Phase 1（`/phase1-interview`）の完了時に `docs/phases.md` が生成され、そこでフェーズ数・内容が確定する。
- Phase 2 以降のフェーズ数・名称・タスクはすべて `docs/phases.md` を正とする。
- エージェントはフェーズ数を事前に想定せず、常に `docs/phases.md` を読んで対象フェーズを判断すること。

### 固定フェーズ（必須）

| Phase | 内容 | 実行方法 |
|---|---|---|
| 1 | 要件定義・仕様策定・工程計画 | `/phase1-interview` |
| 最終 | 成果物の最終検証（動作確認・品質チェック・納品確認） | `/run-phase N` |

- **Phase 1 と最終検証フェーズは必ずフェーズ構成に含める**こと。
- 最終検証フェーズは `docs/phases.md` の最後のフェーズとして配置する。
- 最終検証フェーズの内容はプロジェクト特性に応じて設計するが、以下を必ず含めること：
  - 成果物の動作確認（仕様書との照合）
  - `docs/requirements.md` のゴール・KPI に対する達成確認
  - 納品・リリース判定

### Phase 2〜最終の1つ前は docs/phases.md で動的に定義

`docs/phases.md` が存在しない場合（Phase 1 未完了）は、Phase 2 以降を実行しないこと。
